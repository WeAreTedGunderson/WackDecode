How A Digital Dead Man Switch Works

A dead man switch is not just a mysterious website or a timer. It needs a trigger, a payload path, and a way to prove authenticity.

1 Heartbeat The operator checks in on a schedule.
2 Trigger Missed check-ins or a public event starts release logic.
3 Delay A grace period allows a cancel signal.
4 Release A key, pointer, manifest, or payload becomes public.
5 Verify Recipients confirm the release came from the real operator.
What would count as strong evidence? A signed message from a known McAfee key, a pre-death hash matched by a post-death payload, a public storage pointer tied to him before death, or a deterministic decoding rule that produces independently verifiable output.

WHACKD Mechanics

WHACKD is the strongest technical artifact because it is public, on-chain, and reproducible. Its DMS connection is still speculative.

Official contract

0xCF8335727B776d190f9D15a54E6B9B9348439eEE

  • Etherscan labels the verified contract as Epstein.
  • getwhackd.org identifies the same address as official.
  • CoinMarketCap lists the same contract.
  • The creation transaction is dated 2019-11-16 07:25:23 UTC.

Why it matters

If a decode exists, the official contract is the most defensible starting point. But every decoding theory must say exactly which transactions count.

The phrase "every 1000th transaction" is too vague for decoding work. The counter behavior depends on contract methods and storage state.
1 Transfer starts A holder calls transfer(to, amount).
2 Counter below 999 Receiver gets 90 percent. Ten percent burns.
3 Counter reaches rollover The full transfer amount burns.
4 Counter resets Normal direct transfer behavior resumes.
Community review in the Whackd/whackd-counter repo says transferFrom does not increment the counter. That means Uniswap-routed activity cannot be treated the same as direct transfer calls without reconstructing the actual call history.

Decoding Plan: Three Options / Goals

These are ordered intentionally. Do not start with puzzles. Start by proving which artifacts deserve decoding.

1 Provenance Map: prove what is real enough to analyze

Goal: Determine whether public evidence coherently connects McAfee, WHACKD, and a specific puzzle or payload.

Do

  1. Create sources.csv with source, archive, date, author, and confidence.
  2. Build a graph of claims, domains, accounts, contracts, wallets, and files.
  3. Mark each link as primary, secondary, repost, inference, or unsupported.
  4. Reject screenshot-only or post-facto claims.

Stop when

  • The source depends on private or leaked data.
  • The chain collapses into unverifiable screenshots.
  • The work starts identifying private wallet owners.
2 WHACKD Blockchain Reconstruction: test the concrete artifact

Goal: Reconstruct the official WHACKD contract state and transaction history to test whether a reproducible key or message derivation exists.

Do

  1. Export all transactions and logs for the official contract.
  2. Decode calldata into transfer, transferFrom, and related calls.
  3. Reconstruct the internal counter from creation onward.
  4. Identify full-burn events and generate deterministic candidate strings.
  5. Test only bounded transforms: hex, ascii, keccak, SHA-256, base64, base58, ENS, IPFS, and Swarm formats.

Stop when

  • The theory requires wallet access, private keys, or credential attacks.
  • Address attribution is unsupported.
  • Decoded output exposes private data, credentials, or keys.
3 Public Puzzle / Swarm-IPFS Lab: test candidate artifacts

Goal: Test the strongest public artifacts as puzzle objects, including theories that transaction data derives a content address, feed topic, or storage pointer.

Do

  1. Preserve public files byte-for-byte with SHA-256 hashes.
  2. Run metadata and file-type checks.
  3. Test public reference formats: IPFS CID, Swarm reference, Arweave ID, Ethereum address, ENS content hash, base64, base58, hex, and ascii.
  4. For Swarm theories, test only public owner plus topic retrieval.
  5. Log every transform and discard subjective pattern matches.

Stop when

  • The work becomes password cracking or private-account recovery.
  • Tooling attempts access-control bypass.
  • The only result depends on subjective interpretation.

Recommended Execution Order

This sequence keeps the work honest: provenance first, chain data second, puzzle work last.

1 Source control Create sources.csv, archive URLs, and confidence fields.
2 Contract reconstruction Fetch ABI/source, export logs, decode methods, rebuild counter.
3 Candidate extraction Generate deterministic strings from full-burn and key event classes.
4 Artifact lab Hash originals, test public formats, document negative results.
Recommended gate: if Option 1 does not produce authenticated public candidate artifacts, stop and report that there is no reproducible public DMS evidence.
